
1. 实验目标

本实验实现一个简化 ReAct Agent，将 LLM/Agent 作为任务编排层，将 angr/claripy 作为符号执行与约束求解工具，对自写格式化字符串漏洞程序进行自动化逆向求解。

目标程序中的核心漏洞是：

printf(input);

该调用把用户输入直接作为格式化字符串使用。实验目标是自动构造一个能进入成功路径的格式化字符串 payload。

2. 工程结构
src/crackme.c：格式化字符串漏洞目标源码。
agent/react_angr_agent.py：ReAct + angr Agent 主程序。
tools/angr-python/python.exe：离线 Python + angr 运行环境。
tools/john/bin/gcc.exe：离线 gcc 编译工具链。
logs/run.txt：完整 ReAct 运行日志。
output/solution.json：结构化求解结果。
run_agent.ps1：一键运行脚本。

运行方式：

.\run_agent.ps1

3. 漏洞程序说明

check_payload中先输出审计前缀，再执行危险调用：

printf("audit: ");
printf(input);

随后程序检查输入是否包含 %x、%p或 %s这类格式化字符串特征。若输入以 %x开头，并且内部 secret 未被破坏，则输出：

Success! Format string path reached.

4. 工具封装

Agent 封装了以下工具函数：

compile_target：调用 tools/john/bin/gcc.exe编译 src/crackme.c，生成 crackme.exe。
discover_strings：扫描目标程序中的关键字符串，提取 Success、Wrong payload、format payload等语义线索。
find_success_avoid：用 angr.Project与 CFGFast加载目标并恢复基础 CFG，同时通过 nm.exe获取 check_payload地址。
solve_with_angr：构造符号 payload 缓冲区，使用 angr call_state尝试直接探索 check_payload返回成功的状态。
solve_semantic_constraints：将格式化字符串成功分支谓词交给 angr 依赖的 claripy.Solver求解最短 payload。

5. ReAct 过程

日志中保留了 5 轮完整闭环，超过实验要求的 3 轮：
1. 编译目标程序。
2. 提取语义字符串，明确 find/avoid 目标。
3. 加载 angr 并构建 CFG。
4. 使用 angr 对 check_payload进行符号状态探索。
5. 使用 claripy 对格式化字符串谓词求解具体 payload。
完整日志位于：
logs/run.txt

6. 求解结果
结构化结果位于：
output/solution.json
结果：

{
  "target": "crackme.exe",
  "solution": "%x",
  "expected_output_contains": "Success! Format string path reached.",
  "react_rounds": 5
}

最终 payload：
%x

7. 思考题

本实验中，LLM/Agent 主要承担编排者角色：它不直接替代符号执行，而是负责理解目标语义、拆分任务、选择工具、解释 Observation，并根据工具反馈调整求解策略。

纯符号执行如果直接从 main和运行时输入函数开始，会受到编译器运行时、库函数和 I/O 建模影响。Agent 可以利用字符串语义判断 Success! Format string path reached.是目标、Wrong payload!是规避点，并把搜索空间收敛到 check_payload的关键输入谓词：payload[0] == '%'且 payload[1] == 'x'。这样可以减少无效路径探索，让 angr/claripy 更快得到可复现 payload。
